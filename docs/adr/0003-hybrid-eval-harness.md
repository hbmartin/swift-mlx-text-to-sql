# Two eval harnesses on purpose: Python for breadth, Swift for truth

The eval harness is split deliberately. The Python/uv harness (in `fine-tuning/`) runs the wide sweeps — model × quantization × adapter × schema-serialization × GCD on/off — plus EX scoring, failure taxonomy, and Claude-as-judge, because iterating on eval features in Python is an order of magnitude faster. But it does not exercise the production path: the app runs MLX-Swift with a Swift-side grammar engine and the app's exact prompts, and runtime/tokenizer/grammar drift between stacks is real. So a thin Swift CLI (`creg-eval-cli`) links the app's own `CREGKit` inference code and re-scores the top finalists before any winner is declared. Deleting the Python harness loses sweep breadth; deleting the Swift CLI loses parity with what actually ships.
